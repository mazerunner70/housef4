# Backup schema v1 (`backup_schema_version`: **1**)

Canonical code constant: **`BACKUP_SCHEMA_VERSION_V1`** in [`db/src/types.ts`](../../../db/src/types.ts). Export implementation: **`DynamoFinanceRepository.exportBackupSnapshot`** in [`db/src/dynamoFinanceRepository.ts`](../../../db/src/dynamoFinanceRepository.ts).

**v1 product rule:** structured **metadata only** — no raw bytes of uploaded bank files. **`RESTORE_LOCK`** and other operational-only rows are **never** included in the file.

**Dates:** all epoch **milliseconds UTC** (numbers), consistent with [`../api_contract.md`](../api_contract.md).

---

## 2. Entity records (common rules)

- Each array element is a **plain JSON object** suitable for round-trip to Dynamo items under **`PK = USER#<user_id>`** after restore materialization (see [`../database/data_model.md`](../database/data_model.md) §8.2).
- **`entity_type`** is included on every exported domain object so restore can discriminate rows.
- **Index attributes** (`PK`, `SK`, `GSI1PK`, `GSI1SK`, `GSI2PK`, `GSI2SK`, and any key starting with **`GSI`**) are **omitted** from the file; restore reconstructs keys from ids and relationships.
- **`user_id`** appears on items where the persisted row stores it (accounts, transactions, transaction files, metrics).

**Sort order in v1 export (deterministic, not semantically required):**

| Array | Order |
|-------|--------|
| `accounts` | `name` ascending (case-insensitive) |
| `transactions` | `date` descending, then `id` ascending |
| `clusters` | `cluster_id` ascending |
| `transaction_files` | `timing.completed_at` descending |

## 9. Referential consistency (restore validation)

For v1, a valid backup intended for restore should satisfy at least:

- Every **`transactions[].transaction_file_id`** references an **`transaction_files[].id`**.
- Every **`transactions[].cluster_id`** (when present) references **`clusters[].cluster_id`**.
- **`transaction_files[].account_id`** references **`accounts[].id`** where imports were bound to an account.

Exact validation rules remain with the restore implementation (**400** on semantic failure).
