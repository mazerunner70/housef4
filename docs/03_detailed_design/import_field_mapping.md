## 1. Goal of import mapping

Imports arrive as **different shapes per bank and format**. The backend must always produce a **small, stable internal row** (date, amount, raw merchant text) that downstream clustering, categorization, and APIs can rely on—regardless of whether the user uploaded CSV, OFX, QFX, or QIF.

---

## 2. Canonical import row (what we map *to*)

Before clustering and rules, each parsed line should normalize to something equivalent to these **import-derived** fields:

| Field | Purpose |
|--------|---------|
| **`date`** | When the transaction occurred; the public API exposes this as epoch milliseconds (UTC)—see [`api_contract.md`](./api_contract.md). |
| **`amount`** | Signed number; match product-wide convention (e.g. negative = outflow, per API contract). |
| **`raw_merchant`** | Verbatim payee / description string used for clustering and the review queue UI. |

Optional but useful for support and debugging: **`source_metadata`** (file row index, original OFX `FITID`, raw snippet)—not required by the MVP UI.

The public **`Transaction`** shape in the API also includes `id`, `cluster_id`, `category`, `status`, `is_recurring`. Those are **filled after** normalization (see next section).

---

## 3. Fields derived after parse (not file columns)

These are **not** read directly from CSV/OFX columns as the user’s “mapping” problem in the happy path:

| Field | How it is obtained |
|--------|---------------------|
| **`id`** | Generated by the system when persisting. |
| **`cluster_id`** | Assigned by clustering / merchant canonicalization. |
| **`category`** | From taxonomy + rules engine and user tagging (`POST /api/rules/tag`). |
| **`status`** | `CLASSIFIED` vs `PENDING_REVIEW` based on confidence. |
| **`is_recurring`** | Heuristics (e.g. repeated amount + timing within the same cluster)—see PRD. |

---

## 4. OFX and QFX (structured mapping)

OFX is standardized; use a **fixed mapping** from statement transaction elements to the canonical row (adjust per OFX version if needed):

| Canonical | Typical OFX source |
|-----------|---------------------|
| **Date** | `DTPOSTED` or `DTUSER` on `<STMTTRN>` — parse OFX date format and convert to your chosen ISO / calendar-date rule. |
| **Amount** | `TRNAMT` — apply one global sign convention (OFX uses signed amounts; align with “negative = outflow” if that is the app standard). |
| **Raw merchant** | Prefer `NAME`; if empty, fall back to `MEMO` or `PAYEE` when present. |

**QFX** is treated as OFX content with a wrapper; the same field mapping applies once the payload is parsed.

No user-facing column picker is required for MVP if the OFX parser is correct.

---

## 5. CSV (variable columns)

Bank CSVs differ (`Description`, `Payee`, `Details`, `Amount`, `Debit`, `Credit`, etc.). Recommended approach:

1. **Parse** the header row (handle BOM and quoting).
2. **Detect** columns with heuristics: fuzzy match header names to *date*, *amount* (or *debit*/*credit*), *payee/description*.
3. **Compute** a single signed **`amount`** when the bank splits debit and credit columns.
4. If confidence is low, **later enhancements**: bank-specific **presets** (“Chase checking export”) or a **one-time column mapping** UI.

Avoid pushing bank-specific column names beyond the normalization layer.

---

## 6. QIF (tag-based mapping)

Map QIF record tags into the same three canonical fields, for example:

| Canonical | Typical QIF usage |
|-----------|-------------------|
| **Date** | `D` — parse per QIF date rules. |
| **Amount** | `T` — apply the same global sign convention as OFX/CSV. |
| **Raw merchant** | `P` (payee); optionally append or prefer `M` (memo) when payee is empty or too generic. |

---

## 7. Recommended ingestion pipeline

Keep a clear boundary between **parsing** and **classification**:

```text
File bytes
  → Format detection (csv | ofx | qfx | qif)
  → Format-specific parser
  → List of normalized rows: { date, amount, raw_merchant [, source_metadata] }
  → Validation (drop or quarantine invalid rows)
  → Enrichment: ids, cluster_id, category, status, is_recurring
  → Persist
```

Internal types might use a name such as **`ParsedTransaction`** or **`StagingRow`** for the pre-enrichment shape so bank-specific details never leak into dashboard or ML code.

---

## 8. Amount sign convention and deduplication

**Sign:** Define once whether negative means money **out** (consistent with current frontend examples) and convert every format to that rule.

**Duplicates:** Exports may overlap (e.g. re-importing OFX after CSV). Mitigate with:

- OFX **`FITID`** when present, and/or
- A deterministic hash of **`date` + `amount` + normalized merchant`**, plus policy for transfers and pending vs posted.

Document the chosen rule next to persistence so imports are idempotent where possible.

---

## 9. Summary

- **Map file-specific fields → only `date`, `amount`, and `raw_merchant` (plus optional metadata).**
- **OFX/QFX/QIF:** deterministic tag/element mapping; **CSV:** heuristics, then presets or user column mapping if needed.
- **Category, cluster, status, recurring** are **downstream** of this mapping—not alternate “columns” in the import file.
